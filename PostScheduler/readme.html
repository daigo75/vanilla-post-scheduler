Post Scheduler for Vanilla 2.0/2.1b

Description

Post Scheduler allows to schedule a Discussion to appear at a later date and/or time. A scheduled discussion will be invisible to all Users, except the following:
  • The User who started the discussion.
  • Super Administrators.
  • Any User with the permission to see all Scheduled discussions.
For more details, please see Plugin's Settings page.

Installation

  • Copy the whole PostScheduler folder in the /plugins folder, which can be found in your Vanilla installation folder.
  • Delete all .ini files from the cache folder and all its subfolders. Cache folder is also in your Vanilla installation folder.
  • Enable the Post Scheduler Plugin.

Upgrade

  • In Vanilla Control Panel, disable the Post Scheduler Plugin.
  • Delete all .ini files from the cache folder and all its subfolders. Cache folder is also in your Vanilla installation folder.
  • Enable the Post Scheduler Plugin.

At the end of the Installation or Upgrade, open the Plugin's Settings page. At the bottom, there should be a section titled Overrides. If all messages are green, then you're good to go. If any of them is red, please read the Troubleshooting section below.

Troubleshooting

Post Scheduler requires the override of some Vanilla Core classes to work correctly. To avoid overwriting Core files, such override is performed at runtime, by loading the classes provided by the plugin in lieu of the original ones. However, Vanilla implements a caching mechanism that, sometimes, ignores the new files and loads the standard ones anyway.

If the plugin doesn't seem to work correctly, please do the following:

  • In Vanilla Control Panel, disable the Post Scheduler Plugin.
  • Delete all .ini files from the cache folder and all its subfolders. Cache folder is also in your Vanilla installation folder.
  • Enable the Post Scheduler Plugin.
  • Check the Plugin's Settings page to see if all override classes are now loaded correctly. If that is not the case, please contact Support (see details below).

Usage

The Plugin provides an intuitive User Interface. Instructions can be found on Plugin's Settings page.

Please note that the Plugin's User Interface has only been tested on Vanilla 2.0 with its default theme. Using a different theme will more likely require some adjustments in the interface.

Handling of Scheduled Notifications

As of version 13.02.25, Post Scheduler will also delay the Notifications that are sent to Users who chose to be notified of new Discussions. When a Discussion is scheduled in the future, the notifications it generates are scheduled with it, and sent when appropriate.

For the above mechanism to work, it's necessary to configure your server to call the Cron script periodically. Such script is provided by the Cron plugin. Please refer to Cron plugin instructions for more information.

Requirements

  • PHP 5.3+
  • Vanilla 2.0/2.1b (see Notes)
  • Logger Plugin (either Basic or Advanced version) 12.10.28 or newer
  • Cron Plugin 13.02.24 or newer

Notes

  • Plugin has been tested on Vanilla 2.1b and it should be compatibile with it. However, since Vanilla 2.1 is still in Beta, compatibility cannot be guaranteed. For the same reason, we provide limited assistance for installations in such environment.
  • Delaying of notifications on Vanilla 2.1b is experimental and could have some issues. This is due to the new mechanism that such version uses to manage notifications, which is still incomplete and contains several bugs.
  • Highlighting of scheduled Discussions has been designed for Vanilla's default theme. Using the plugin with 3rd party themes might require some adjustment to the plugin's CSS files.
  • All dates and times displayed by the plugin on the frontend are in current User's timezone. That is, if a Post is scheduled for 00:00 GMT, an Administrator in New York will see the it scheduled for 7:00 PM.
  • Important: this plugin requires some modifications to Vanilla Core files. Such changes are made automatically, in a non-permanent way when the plugin is enabled, and don't affect normal operations even when the plugin is disabled.
    If you wish to ensure that every modification to Core files is removed when the plugin is disabled, simply remove all ini files from cache folder and its subfolders.
